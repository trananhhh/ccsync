# Phase 2 — `.ccsyncignore` per code folder

## Overview

`.stignore` is first-match-wins (Syncthing, not git), so per-folded source order *is* precedence order. Today ccsync compiles `.stignore` from two layers — `GLOBAL_IGNORE_PATTERNS` + `cfg.globalIgnore`, then `bucket.ignore[]`. This phase adds a third, per-project layer: a `.ccsyncignore` file at the root of each synced code folder. It is the LAST authority in the compiled file, so user intent always wins, and it lives next to `.gitignore` so it's discoverable and therefore git-committable.

Stays inside one schema (`z.record(z.string(), ...)` already accepts arbitrary keys — no migration). Scope: 1 new core file, 3 modified core files, 1 new test file, 1 README section.

## Entry state

- `src/core/ignores-default.ts:buildStignore(bucketIgnore, globalIgnore)` writes a 2-layer compiled file (global section → bucket section).
- `src/core/stignore-writer.ts:WriteStignoreInput = { folderPath, bucket, globalIgnore }` — writes `.stignore` into `folderPath` for every entry produced by `applier.collectStignoreTargets(cfg)`.
- `src/core/applier.ts:collectStignoreTargets()` iterates `cfg.buckets`; for the `code-root` bucket it pushes one `StignoreTarget` per `bucket.paths` entry.
- No per-folder ignore file is consulted. Users who want per-project ignore patterns must edit `cfg.buckets['code-root'].ignore[]` (hard to discover) or hand-edit `.stignore` (gets clobbered on next apply).

## Exit state

- A new `src/core/ccsyncignore.ts` exports `readCcsyncignore(folderPath): Promise<string[]>`:
  - reads `<folderPath>/.ccsyncignore` (UTF-8);
  - splits on `\r?\n`;
  - trims trailing whitespace;
  - drops blank lines and lines that start with `//`;
  - returns patterns unchanged otherwise (verbatim — ccsync does NOT resolve them).
- `src/core/ignores-default.ts:buildStignore(bucketIgnore, globalIgnore, projectIgnore?: string[])` gains an optional third argument. When supplied, emitted as a fourth section `// Project (.ccsyncignore) — <folder>` after the bucket section, in exact source order. Empty array behaves identically to `undefined` (no section).
- `src/core/stignore-writer.ts:WriteStignoreInput` gains `codeFolderRoot?: string`. When present, the writer reads `.ccsyncignore` from there and passes the lines to `buildStignore`. When absent, the existing 2-layer compile runs unchanged.
- `src/core/applier.ts:collectStignoreTargets()` tags every `code-root` target with `codeFolderRoot: <bucket.paths entry>` (the folder root itself). Other buckets leave the field undefined.
- `buildStignore` prepends `#escape=\` to the compiled output iff the merged `.ccsyncignore` payload contains any backslash character (Windows-on-Windows cross-platform projects).
- A new `findUnanchoredNegations(projectIgnore: string[]): string[]` helper in `src/core/ccsyncignore.ts` returns lines that start with `!` after trimming but are not root-anchored (i.e. don't begin with `!/`). The `apply` flow logs a `soft warning` for each via `log.warn(...)`. Users can read `<folder>/.stignore` themselves to see the warning targeted at the offender.
- `src/cli/commands/setup.ts:configureRootProfile` and `src/cli/commands/join.ts` print a single `log.plain` line after the picker: "I scanned your code folders; I found a `.ccsyncignore` in N of them — they'll be applied automatically."
- README has a new **Ignore rules** subsection explaining: (a) the merge order (global → bucket → `.ccsyncignore`); (b) the first-match-wins foot-gun; (c) `#` is NOT a comment in `.stignore`; (d) root-anchored negations (`!/path`) avoid the `cache/` traversal trap.

## Demo

A user commits `.ccsyncignore` at the root of their Next.js project:

`/Users/me/work/my-saas/.ccsyncignore`:

```
// Don't sync huge lockfiles — regenerate locally
pnpm-lock.yaml
package-lock.json
yarn.lock

// Keep coverage output — useful across machines
!/coverage/lcov.info
!/coverage/coverage-summary.json

// Next.js scratch
.next/cache

// Per-machine editor cruft
.vscode/*
!.vscode/launch.json
!.vscode/settings.json
```

`ccsync apply` produces `/Users/me/work/my-saas/.stignore` (excerpt — the actual file will contain all 30 GLOBAL_IGNORE_PATTERNS upstream):

```
// Generated by ccsync — do not edit manually

// Global protections
.git/index
.git/index.lock
… (29 GLOBAL_IGNORE_PATTERNS entries) …

// User global (cfg.globalIgnore)

// Bucket: code-root

// Project (.ccsyncignore) — /Users/me/work/my-saas
pnpm-lock.yaml
package-lock.json
yarn.lock
!/coverage/lcov.info
!/coverage/coverage-summary.json
.next/cache
.vscode/*
!.vscode/launch.json
!.vscode/settings.json
```

Visible behaviour:
- No warning is printed, because every negation is root-anchored.
- If a developer edits `.ccsyncignore` to drop the leading `/` on one of those `!` lines, the next `ccsync apply` prints: `⚠ .ccsyncignore: unanchored negation "!coverage/lcov.info" — Syncthing may force directory traversal; prefix with "/" to root-anchor`.
- Empty `.ccsyncignore` is equivalent to missing — no `// Project` section is emitted.

## Implementation Steps

1. **`src/core/ccsyncignore.ts`** (new): export `readCcsyncignore(folderPath): Promise<string[]>` (`fs.readFile` UTF-8 + split `\r?\n` + trim trailing whitespace + drop `""` and `//`-prefixed lines, verbatim otherwise); `findUnanchoredNegations(lines: string[]): string[]` (matches `^!\s*\S`, excludes `^!/`); `usesBackslash(lines: string[]): boolean`.
2. **`src/core/ignores-default.ts`**: signature becomes `buildStignore(bucketIgnore: string[], globalIgnore: string[] = [], projectIgnore?: string[]): string`. When `projectIgnore` is non-empty, append `// Project (.ccsyncignore) — <folder>` then the lines verbatim. Prepend `#escape=\` iff `usesBackslash(mergedLines)` is true. When undefined or `[]`, today's 2-layer output is unchanged.
3. **`src/core/stignore-writer.ts`**: extend `WriteStignoreInput` with `codeFolderRoot?: string`. Treat as the absolute folder root — read `.ccsyncignore` from there; ENOENT ⇒ ignore silently.
4. **`src/core/applier.ts:collectStignoreTargets(cfg)`**: for `name === "code-root"` entries push `{ folderPath, bucket, codeFolderRoot: folderPath }`. Other buckets leave the field undefined. In `apply()`, after `writeStignore(...)`, run `findUnanchoredNegations(projectIgnore)` and `log.warn(...)` per offending line (read `.ccsyncignore` once — pass the result back from the writer, do not re-read).
5. **`src/cli/commands/setup.ts:configureRootProfile`** and **`src/cli/commands/join.ts`**: count picked folder roots that contain `.ccsyncignore`; if `> 0`, print `log.plain("I scanned your code folders; I found a .ccsyncignore in N of them — they'll be applied automatically.")`.
6. **`README.md`** — append **Ignore rules** section: merge order, first-match-wins warning, `#` is not a comment, root-anchored-negation recipe, the hand-traced Next.js example mirroring the Demo above.
7. **Tests**: `tests/core/ccsyncignore.test.ts` (new) plus the three extends below.
8. **Verification**: `pnpm typecheck && pnpm lint && pnpm test && pnpm build`. Manual smoke: write `.ccsyncignore` in `/tmp/ccsync-ignore-smoke`, run `ccsync apply`, assert the produced `.stignore` matches the Demo.

## Risk Assessment

(Pulled from `approach.md` § Risks.)

| Risk | Severity | Proof |
|---|---|---|
| `.ccsyncignore` semantics confusion (stignore ≠ gitignore, first-match-wins) | MED | README "Ignore rules" subsection; manual smoke (Negation demo): apply, read `.stignore`, hand-trace Syncthing's behaviour on a test bench. |
| Negation traversal (`!cache/` scans the whole tree) | MED | `findUnanchoredNegations` warning at apply time. Unit test covers detection. |
| Bucket ignores overridden by `.ccsyncignore` re-emit | LOW | Compile order is unchanged. Unit test on `tests/core/ignores-default.test.ts` asserts section order. |
| `#`-prefixed lines treated as pattern (stignore rejects) | LOW | Parser rejects with a soft warning (emit a `log.warn` and treat the line as a stignore pattern verbatim, as documented in research §5.2). Unit test covers detection. |
| Symbol-globs that include dots in the deck — `.syncthing.*` already auto-ignored by Syncthing | LOW | No action needed (Syncthing's own auto-ignore). Documented in README. |

## Security Considerations

- `.ccsyncignore` is opt-in (user creates the file). Parser does not glob-regex the lines — ccsync emits them verbatim, so this is `text-in / text-out`. No eval, no template-injection surface; Syncthing consumes the compiled `.stignore` natively.
- One extra `fs.readFile` per code-root on each `ccsync apply` (UTF-8, bounded by one file per folder). No path traversal: `codeFolderRoot` always comes from `cfg.buckets['code-root'].paths`, set in `setup.ts` after `pickCodeFolders` resolves candidates against the user's chosen root.
- Soft-warning channel (`log.warn`) prints the user's own line — no redaction needed because the line is the user's content.
